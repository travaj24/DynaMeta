# DynaMeta full codebase audit (2026-06-07)

Method: a 7-lens multi-agent audit (optics-physics, carrier-physics, bugs/robustness, standardization, organization/quality, compatibility, gaps/flexibility/speed). Every high/critical finding was ADVERSARIALLY VERIFIED against the actual code (a separate verifier agent tried to REFUTE each). 37 raw findings -> 8 confirmed, 5 refuted as false positives. 22 agents.

## Executive summary

DynaMeta is a well-architected, validation-driven multiphysics library (carriers -> bridge -> optics) for tunable metasurface modulators. Overall health is strong: the seam-based design (LayeredStackSolver/EffectModel/NToEpsMap protocols), lazy-loaded optional heavy solvers, and extensive cross-solver validation (TMM/FEM/FDTD vs analytic oracles) reflect mature engineering. The audit surfaced no silent-correctness landmines in the shipping numerical core that survived adversarial verification; most refuted findings were false alarms about optional-dependency imports (already correctly lazy-loaded via pipeline.py and PEP-562 __getattr__) and a 3D-solver complex-index guard that actually fails loud (TypeError in max()) rather than silently.

Three themes dominate the real findings. (1) Robustness gaps at input boundaries: a genuine NoneType crash in fit_drude_lorentz when all multi-start fits fail (fdtd_seam.py:161), an unguarded divide in sheet_resistance_ohm_sq (analysis.py:91), and a hard module-level devsim import (devsim_layered.py:25) that breaks the optional-dependency contract for anyone who imports the module directly. (2) Defensive-guard consistency: the FDTD 2D/3D solvers accept n_super/n_sub but never validate losslessness the way the FEM path does (solver.py:158-165), so a complex index produces an unhelpful TypeError instead of the documented NotImplementedError. (3) Capability/performance gaps in the optical stack: no RCWA backend despite the protocol being reserved, no Lorentz ADE in 3D FDTD, no oblique/non-vacuum FDTD for structured cells, and a per-wavelength FDTD re-settle in run_pipeline that is 10-100x slower than the already-written broadband path (run_fdtd_sweep). These last items are the strategic frontier: closing them turns DynaMeta from a characterization tool into a fast inverse-design platform. The carrier-physics findings are mostly documented scope limitations (no recombination in single-carrier DD, parabolic SP Fermi level, hard-wall dead layer) rather than bugs, and should be addressed with guards/docs, not rewrites.

## Strengths
- Seam-based extensible architecture: LayeredStackSolver, EffectModel, and NToEpsMap protocols (core/interfaces.py) let users bring their own carrier/optical solvers; the TMM reference (tmm_reference.py) proves the pattern end-to-end.
- Disciplined optional-dependency hygiene at the package boundary: heavy solvers (ngsolve/netgen/devsim) are lazily imported in pipeline.py and via PEP-562 __getattr__, so `import dynameta` and pure TMM/numpy paths run without the [solvers] extra (5 of the 6 'unguarded import' findings were refuted on this basis).
- Strong cross-solver validation culture: FDTD reduces to TMM, oblique-vs-TMM, FDTD-2D-vs-RCWA, Drude+Lorentz-vs-dispersive-TMM, and DEVSIM-DD-vs-analytic oracles, with the lossless-trap pitfall explicitly guarded by per-order checks rather than energy tripwires.
- Physically careful FDTD engine: impedance-matched CPML per end medium, homogeneous-superstrate incident reference run for exact reflection subtraction, Snell flux ratio in T0, and a validated Drude+Lorentz ADE in 2D-TE.
- Centralized physical constants (constants.py) as a single source of truth to prevent drift between optics and carrier solvers.
- Honest, well-documented scope limits: single-carrier DD explicitly states 'no recombination', the SP solver documents its hard-wall dead layer and parabolic-Fermi assumptions, and the FEM solver raises a clear NotImplementedError on lossy end media.

### Missing validation of complex (lossy) superstrate/substrate indices in FDTD 2D solver  [high -> high]
- **Where:** `dynameta/optics/fdtd_nd.py:394-420`
- **Issue:** The solve_fdtd_2d function accepts n_super and n_sub as parameters and fills the z-pad regions with eps_inf = n_super^2 and n_sub^2 (lines 419-420). If a caller passes complex n_super or n_sub with non-zero imaginary part, the eps_inf assignment becomes complex. This violates the Yee solver's assumption of REAL permittivity and causes SILENT WRONG RESULTS: the E-field normalization in the semi-implicit Drude ADE becomes inaccurate, and the overall R/T extraction becomes physically incorrect. The FEM solver (solver.py:158-165) correctly RAISES NotImplementedError with a clear error message on lossy superstrate, preventing silent failure. The FDTD 2D/3D solvers should mirror this guard.
- **Fix:** Add explicit validation at the entry point of solve_fdtd_2d (after line 393) and solve_fdtd_3d (after line 937): raise NotImplementedError if abs(complex(n_super).imag) > 1e-9 or abs(complex(n_sub).imag) > 1e-9, matching the guard in solver.py:158-165. Include the same explanation: 'R/T/A and energy budget are defined only for LOSSLESS end media'.
- **Verifier:** The audit finding is REAL. The solve_fdtd_2d and solve_fdtd_3d functions accept n_super and n_sub parameters and directly use them to set eps_inf = n_super^2 and n_sub^2 (lines 419-420 and 966-967 respectively) without validation that these indices are lossless (real). While the docstrings correctly state the requirement for lossless indices, there is no runtime validation. If a caller passes complex n_super/n_sub with non-zero imaginary parts, the code will fail with an unhelpful TypeError at line 400 (in max() comparison) rather than a clear NotImplementedError explaining the actual semantic constraint. This is inconsistent with the FEM solver (solver.py:158-165) which DOES raise NotImplementedError with a clear message. The fdtd_seam.py wrapper does validate (lines 300-303) but direct calls to the public solve_fdtd_2d/3d functions bypass this protection. The risk of silent wrong results is low due to the early TypeError, but the missing explicit validation is a code quality/consistency issue that mirrors the FEM approach and matches the audit recommendation.

### Lack of recombination model in single-carrier DD (electron-only) path  [high -> low]
- **Where:** `dynameta/carriers/physics_drift_diffusion.py:14-16`
- **Issue:** The drift-diffusion setup includes a continuity equation for electrons but NO recombination term (no SRH, radiative, or Auger). The continuity equation is div(J_n) = 0 (no recombination, line 15 docstring). For a gated accumulation-only device this is fine, but if a transient or AC sweep ever injects/removes carriers or runs a long transient, the absence of recombination means carriers pile up indefinitely with no decay mechanism. This is not a silent bug (the physics is explicit), but it is a significant limitation for any dynamics beyond the quasi-static gate sweep.
- **Fix:** Document clearly in the docstring and the TransportModel that the single-carrier DD model assumes NO recombination and is valid only for quasi-static/steady-state gating without injection. Add a recombination term (tau_srh from TransportModel.tau_srh_s) to the continuity equation so transient solves become physically meaningful. A placeholder (todo/phase4) note is acceptable if this is a known future work item.
- **Verifier:** The finding is technically accurate: the single-carrier DD module (physics_drift_diffusion.py) contains NO recombination term in the continuity equation (no SRH, radiative, or Auger mechanisms). However, this is NOT a bug or oversight???it is an intentional, documented design choice appropriate for the module's scope. The module's docstring explicitly states (line 15) "Continuity: div(Jn) = 0 (DC, no recombination)" and line 139 acknowledges "No recombination in this model -> no node_model on the continuity equation." The module targets quasi-static gating of gated capacitors (gate | oxide | semiconductor | ohmic) where carriers have fixed boundary conditions at contacts and the device is in steady state or accumulation???no competing generation/recombination. For devices requiring carrier dynamics, SRH recombination, or transient behavior, the project provides a parallel bipolar+SRH formulation (physics_bipolar_dd.py, lines 163-169) which is used in other validation tests (ac_diode.py, transient_diode.py). The limitation is neither silent nor hidden; it is core to the intentional scope. The recommendation to add recombination conflicts with the architecture: this module is meant to be focused for its use case, with the full physics available in the bipolar module.

### File-descriptor redirection (os.dup/os.dup2) not portable to Windows  [high -> medium]
- **Where:** `dynameta/validation/ac_diode.py:62-66; ac_capacitance.py:61-72; similar in bipolar_diode.py, gated_dd_2d.py, fdtd_devsim_topology_design.py`
- **Issue:** Multiple validation scripts use os.dup()/os.dup2() to suppress stdout during DEVSIM solver iterations. - **Verifier:** The code does use os.dup()/os.dup2() for stdout redirection in 10 validation scripts. While these functions technically execute without error on Windows (verified on Windows 11), the finding's core claim about Windows portability concerns is REAL but the severity is overstated. The functions work at the Python level, but POSIX file descriptor semantics differ from Windows, and C-level libraries may not respect the redirection consistently. However, since: (1) these are validation scripts run manually where solvers are available, (2) CI runs only on ubuntu-latest, (3) the code doesn't crash, and (4) the library documentation does not claim Windows support, this is a MEDIUM severity concern (not HIGH) -- it's a legitimate compatibility issue that would prevent the validation suite from working reliably on Windows, but doesn't affect the core library's API or functionality. ### Per-wavelength FDTD re-solve redundantly settles the time-domain pulse tail at every wavelength  [high -> high]
- **Where:** `dynameta/pipeline.py:127-132`
- **Issue:** The optical solve loop invokes `solve_optics(design, geo, eps_by_region, lambda_m, ...)` once per wavelength in the sweep. For the FDTD backend (make_fdtd_optical_solver), each wavelength call runs a FRESH 2D/3D Yee solver on the [lambda_m*(1-band_frac), lambda_m*(1+band_frac)] band with its own CPML settling time (controlled by `settle` parameter, default 12 tau, line 284 in fdtd_seam.py). This means a 10-wavelength sweep at a fixed bias runs ~10 full settling transients instead of 1. The faster path exists (run_fdtd_sweep in fdtd_seam.py:419-441) which does ONE broadband FDTD and interpolates all wavelengths, but it is not integrated into run_pipeline's optical_solver seam -- users must hand-roll a replacement optical_solver or manually call run_fdtd_sweep before the pipeline. For dispersive materials or Lorentz poles, the per-wavelength tail settling is 10-100x slower than necessary.
- **Fix:** Expose run_fdtd_sweep's per-bias-fast-path pattern as a first-class optical_solver builder (e.g. make_fdtd_sweep_optical_solver) that wraps fdtd_sweep_spectrum and interpolates. Alternatively, detect wavelength-sweep context inside make_fdtd_optical_solver and auto-switch to the broadband path when multiple wavelengths are in flight. This is a medium-lift fix that closes a known bottleneck mentioned in the fdtd_seam docstring itself (line 424).
- **Verifier:** The finding is genuine. The `run_pipeline()` function (dynameta/pipeline.py:127-132) calls `solve_optics()` in a nested loop over both bias points and wavelengths. When `optical_solver` is set to `make_fdtd_optical_solver()` (dynameta/optics/fdtd_seam.py:282-344), each wavelength invokes a complete FDTD solve with full CPML settling (default 12 tau on line 283, applied within a band_frac window around each lambda_m on line 314). The faster alternative `run_fdtd_sweep()` (lines 419-441) does ONE broadband FDTD and interpolates all wavelengths, explicitly documented as "the fix for the per-wavelength seam re-running the full settling tail at every wavelength" (line 424). However, `run_fdtd_sweep()` is not integrated into the default optical_solver seam and must be manually invoked or wrapped by users. There is no auto-detection in `make_fdtd_optical_solver()` to switch strategies when multiple wavelengths are in flight. A 10-wavelength sweep therefore incurs ~10 redundant settling transients instead of 1, making the default path 10-100x slower than necessary for dispersive materials. The issue is not a misreading, false positive, or already guarded???it reflects a genuine architectural gap in the library's public API.

## Feature proposals

### RCWA optical backend (Rigorous Coupled-Wave Analysis)  [transformative value / large effort]
- **Rationale:** The codebase explicitly reserves LayeredStackSolver protocol for RCWA (core/interfaces.py:90-95). TMM is exact for uniform stacks but fails for laterally-patterned gratings. FEM handles patterned structures but is expensive for high-frequency designs; RCWA provides the missing middle: exact semi-analytical Fourier-modal solution for 1D/2D patterned metasurfaces at a fraction of FEM's cost. Given the library's focus on tunable metasurface modulators with patches/gratings, this would unlock fast parametric sweeps and inverse design over structured devices (fdtd_seam.py already rasterizes inclusions but FDTD is broadband-expensive for narrow-band designs).
- **Sketch:** Implement RCWASolver(LayeredStackSolver) wrapping a port of S4 or grcwa, reusing the LayeredStack abstraction (core/layered.py already structures graded layers for this). Plug into run_pipeline(optical_solver=...) alongside tmm_reference.py's make_layered_tmm_solver(). Validation: cross-check against FEM on a laterally-uniform limit and against FDTD on structured cells. Payoff: O(100x) speedup over FDTD for grating R(lambda) sweeps, enabling adjoint inverse design on realistic patterned devices.

### Plotting/visualization layer for results  [high value / medium effort]
- **Rationale:** The library orchestrates complex multiphysics pipelines (carriers -> bridge -> optics) but has NO visualization. Users must manually extract OpticalResult.R/T/A and SweepRow arrays to plot. No built-in (lambda vs R, bias vs modulation, topology-optimized density fields, electric field maps). Examples (validation/_reference_device.py) run sweeps but don't show outputs. Tests validate numerics but don't expose results. This friction hampers exploration and slows validation feedback. Matplotlib is a lightweight dependency (already common in scientific Python).
- **Sketch:** Create dynameta/result_plotter.py with functions: plot_spectrum(sweep_rows, ylabel='R') for 1D reflection/transmission vs wavelength over bias states; plot_bias_contrast(sweep_rows, lambda_nm) for modulation depth vs gate bias; plot_density_field(rho, title='Topology') for 2D projected-density visualizations; plot_e_field_amplitude(e_field_grid, plane='xy'). Expose OpticalResult.A_independent alongside energy-budget R+T plot. Thread through examples/ and validation/ so users see quick visual feedback.

### Adjoint/gradient-based inverse design for structured metasurfaces  [transformative value / large effort]
- **Rationale:** Topology optimization (optics/topology_opt.py) exists over JAX-FDTD but is limited to broadband Drude material + chi3 nonlinearity. Adjoint sensitivities are a first-class capability in gradient-based optimization libraries (Meep, Lumerical, Tidy3D). DynaMeta has: (1) JAX autodiff backend in fdtd_nd.py; (2) density_filter + project framework in topology_opt.py; (3) Adam optimizer in inverse_design.py. Missing: (a) RCWA adjoint (pairs with RCWA backend above) -- needed for narrow-band designs where FDTD's numerical dispersion dominates; (b) parameterized geometry (polygon vertices, radii) inverse design beyond density, so users can push past binary-density artifacts; (c) multi-wavelength/multi-objective fidelity (e.g. maximize contrast at lambda1, minimize loss at lambda2). This is the capability that transforms the library from a characterization tool to a design tool.
- **Sketch:** Add optics/rcwa_adjoint.py with jax.grad-compatible RCWA kernel (leveraging autograd-compatible S4 or custom JAX layers). Extend topology_opt.py to accept objectives like 'maximize_R_at_lambda1 - alpha*loss_at_lambda2'. Add optics/parametric_design.py where users define spline/polygon patches via continuous parameters, then optimize those directly (vertices, heights, radii) rather than pixel densities. Pair with a user-facing API: design_opt = ParametricInverseDesign(fom_fn=..., params0=..., bounds=...); result = design_opt.optimize(). Validation: reproduce published topology-optimized metasurface designs (e.g., Yao et al. passive/active topos).

### Coupled carrier-optics transient solver (thermo-optic + dynamic modulation)  [high value / large effort]
- **Rationale:** The codebase has isolated transient pieces: carriers/transient.py (time-stepping), optics/fdtd.py (time-domain EM), thermal drivers (thermal.py). But they are NOT coupled: run_pipeline enforces STEADY-STATE (DC carriers + CW optics). A tunable modulator under a voltage pulse or thermal transient involves: E(t) -> n(t) [carrier lifetime / RC dynamics] -> eps(t) [Drude free-carrier / thermo-optic] -> R(t) [optical transient]. Today users must manually loop carriers.transient.solve_time_evolution -> bridge.assemble_eps(t) -> fdtd.solve(t), losing orchestration benefits. Thermal modulators add temperature feedback (Joule -> T -> eps_thermo -> R), risking runaway without validated thermal-optics co-sim.
- **Sketch:** Create carriers/coupled_transient.py: drive carriers.transient.solve_time_evolution + thermal_fem.solve_thermal_transient + effect-model bridge in a synchronized loop. FDTD furnishes power dissipation (volumetric Im(eps)|E|^2) back to the thermal solver. Pipeline gains time_evolution(design, bias_pulse, t_span, t_steps, ...) returning (t, R(t), T(t), n(t)). For thermal-optic modulators, validate against measured ringdown / response time. Pairs with a future carrier-optical feedback loop for all-optical modulation (chi3 Kerr -> intensity -> modulation).

### Distributed caching & result serialization / HDF5 export  [high value / medium effort]
- **Rationale:** Sweeps are expensive: a 10x20 (bias x wavelength) grid on a 3D FEM mesh with 10k+ DOFs can take hours. Today users must re-run the full pipeline on each iteration (design tweak, bias point added, wavelength refinement). No built-in checkpointing or incremental solve. No standard export format for downstream tools (machine learning, Lumerical, COMSOL, plotting libraries). test_bridge.py and validation scripts run small grids; scaling to real device sweeps lacks infrastructure.
- **Sketch:** Add dynameta/cache.py: simple SQLite or HDF5 backend that caches OpticalResult per (design_hash, bias, lambda_m, solver_config). run_pipeline() gains cache= kwarg: if hit, skip solve; if miss, save result. Add dynameta/export.py: sweep_to_hdf5(rows, fname), sweep_to_csv(rows, fname) for pandas/matplotlib/Jupyter analysis. Validation: measure wall-time for a 100-point sweep with/without cache. Payoff: iterate designs at interactive speed; integrate with Jupyter notebooks for fast exploration.

### Magneto-optic (Faraday) tensor optical solver with circulators  [medium value / large effort]
- **Rationale:** The codebase reserves off-diagonal tensor eps for future work (fdtd_nd.py:28 'DEFERRED: off-diagonal/magneto-optic tensor eps', fdtd_mo.py exists for 1D magneto-optic anisotropy). Magneto-optic modulators (Faraday rotation in YIG/garnet on ITO) are a separate physical modulation mechanism (no free-carrier / thermal / EO). solver.py currently assumes Hermitian eps; magneto-optic needs non-Hermitian (gyrotropic) response. This opens a new device class (isolators, circulators, gyrators) beyond the current free-carrier/EO family.
- **Sketch:** Extend core/effects.py with GyromagneticEffect (magnetization -> off-diagonal eps tensor). Extend fdtd_nd.py 2D/3D kernels to handle full 3x3 complex tensor eps (currently diagonal). Validate 1D against fdtd_mo.py, then 2D (Faraday slab). Add solver.py support for non-Hermitian eps in the FEM formulation (the curl-curl operator remains Hermitian, but BC+material handling must adapt). Integration point: assemble_eps(..., extra_fields={'magnetization': ...}) threads through EffectEpsMap as for Pockels/Kerr. Payoff: unlocks magneto-optic modulator design, a complementary family to ITO-gate architectures.

### Fast convergence harness (automated grid refinement & cross-solver validation)  [high value / medium effort]
- **Rationale:** Validation scripts (fdtd_2d_reduces.py, oblique_vs_tmm.py) manually hard-code mesh/time-step parameters and compare solvers. No automated convergence sweeping (dx, dt, mesh order, PML size -> R error). Users must manually trial-and-error grid parameters to balance accuracy vs speed. The library ships FOUR optical solvers (TMM, FEM, FDTD, future RCWA) but no unified framework to verify they agree. For a new design or material, users cannot ask 'is my mesh fine enough?' without manually re-running 10 parameter combinations.
- **Sketch:** Create validation/convergence_harness.py with functions: validate_against_oracle(design, sweep, oracle_solver, test_solver, params_to_vary={"dx": [2e-9, 1e-9, 0.5e-9], ...}) that sweeps FDTD/FEM grids, measures R(param) convergence, plots residuals, returns the minimum parameter set for 1e-4 error. Extend tests/ with a suite that runs this harness on standard test cases (uniform slab, grating, patch). Payoff: users gain confidence in new designs; design space exploration is faster (users know smallest grid that meets their tolerance).

### Multi-material field-effect effects (Pockels on GaAs, LiNbO3 crystal angles, EO polymer switching)  [medium value / medium effort]
- **Rationale:** core/effects.py implements PockelsEffect generically (E -> r_ijk tensor -> birefringence). But material-specific optics.eo_models would allow users to plug in measured electro-optic coefficients for GaAs, LiNbO3, PLZT, KTP, EO polymers with minimal boilerplate. Today: users hand-code r_coefficients. No library of material EO tensors (like materials/db.py for Drude metals). This friction blocks rapid exploration of EO-active material platforms (III-V semiconductors, ferroelectrics, polymers).
- **Sketch:** Extend materials/optical_model.py with EoOpticalModel(r_tensor, ne_n0_lambda, ...) for anisotropic crystal platforms. Build materials/eo_materials.py bundling published r tensors for GaAs, LiNbO3, KTP, PLZT, EO-PMMA. Add effects.py integration: EffectModel can read material's r_ij directly. Pair with core/crystal_orientation.py (angle -> rotated tensor) so users can twist the crystal and see birefringence Bragg tuning. Validation: measure LiNbO3 EO tuning curves, match published data.

### Mode solver / port definition for waveguide coupler designs  [medium value / large effort]
- **Rationale:** The library focuses on reflection/transmission modulators (normal-incidence optical). Waveguide couplers (evanescent / directional / Mach-Zehnder modulators) require: (a) mode solver (find guided modes, effective index); (b) port/mode definitions (inject mode n_eff into FEM/FDTD); (c) coupling coefficient extraction (overlap integral). solver.py and fdtd_nd.py assume plane-wave excitation. A mode-solver seam would unlock photonic-integrated-circuit designs (PIC modulators, switches on SOI / SiN). Third-party tools (Lumerical MODE, COMSOL) solve this, but DynaMeta's end-to-end Drude bridge is a competitive advantage if it can thread through to waveguides.
- **Sketch:** Add optics/mode_solver.py: wrap ngsolve (curl-curl eigenvalue problem on a cross-section) to find mode fields + n_eff. Create core/port.py (mode definition with index, amplitude, phase). Extend fdtd_nd.py to inject modes: _run_2d_te_with_port(mode_field, ...) synthesizes the soft source from the mode shape. Extend solver.py to support port boundary conditions (absorbing/matched at mode ports). Payoff: simulate directional couplers, Mach-Zehnder arms, phase shifters; validate against benchmarks (3dB coupler length, isolation).

## Top features (synthesis pick)

- **RCWA optical backend (RCWASolver implementing LayeredStackSolver)** -- Highest-leverage addition: the protocol is already reserved (core/interfaces.py:90-95) and the LayeredStack/LayeredSlab model + slice_eps_field are validated. RCWA fills the gap between exact-but-uniform TMM and expensive 3D FEM, giving exact semi-analytical Fourier-modal R/T for laterally-structured slabs with lossy claddings (the exact case FDTD currently rejects at fdtd_seam.py:301-313) at 10-50x FDTD's cost. Unblocks fast parametric sweeps for the project's core metasurface-grating use case.

Adding an RCWA adjoint (pairs with the RCWA backend), parameterized-geometry optimization (polygon/radii beyond pixel density), and multi-wavelength/multi-objective FOMs (maximize contrast at lambda1, minimize loss at lambda2) is the capability that makes the whole stack a design tool, with the differentiable seam already proven.
- **Lorentz ADE in 3D FDTD + oblique/non-vacuum structured FDTD** -- Removes two confirmed capability walls: solve_fdtd_3d rejects Lorentz poles (fdtd_nd.py:938-940), forcing dispersive 3D structures to drop to 2D or frozen-eps; and the FDTD seam can't do oblique or non-vacuum end media on structured cells. - **Results visualization + caching/HDF5 export layer** -- Large quality-of-life multiplier with modest effort. - **Coupled carrier-optics transient solver (dynamic/thermo-optic modulation)** -- Opens a new device class.
